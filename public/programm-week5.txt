## Слайд 6
**CLAUDE.md**
Память проекта

CLAUDE.md

Claude не помнит ничего между сессиями. CLAUDE.md — файл в корне проекта, который он читает с первой секунды. Без него — каждый раз объясняешь с нуля.

/init

Команда /init

Claude читает папку проекта и сам генерит базовый CLAUDE.md. Стартовая точка — не пустой файл.

Руки

Ты правишь вручную

Добавляешь стек, правила, структуру папок. Чем точнее — тем меньше вопросов от Claude.

Ошибки

После каждого косяка

Claude сделал что-то не так → добавляешь правило «никогда не делай X». CLAUDE.md растёт из живого опыта.

Включи Memory в настройках

claude.ai/settings → Memory → ON — Claude начнёт запоминать контекст между разговорами.

Глобальный файл

~/.claude/CLAUDE.md — правила для всех проектов сразу.
Например: "всегда пиши на TypeScript, никогда не трогай .env"

## Слайд 8
**5 правил промпта**
Стандарты проекта

5 правил в CLAUDE.md

Копируй в свой CLAUDE.md — Claude будет следовать им в каждой сессии без напоминаний.

1

Документация и тесты

Без архитектурного дока следующий Claude строит на песке. Без теста — не знаешь, что сломал.

MUST: перед кодом — создай
/docs/ARCHITECTURE.md и PRD.md.
После каждой функции — unit-тест:
happy path + 1 edge case.
MUST NOT: коммитить без теста.

2

Файл ≤ 500 строк

Claude теряет контекст в длинных файлах — начинает угадывать то, что выше видимой области.

MUST: если файл → 500 строк,
останови и разбей на модули
до продолжения.
MUST NOT: файлы >500 строк
в новом коде.

3

Claude Opus 4.6 — лучшие практики

С /think Opus решает на уровне senior. Без него — junior. Та же цена, другой результат.

Для сложных решений используй
extended thinking: напиши /think
перед задачей.
Один промпт = одна задача.
XML-структура для длинных промптов.

4

Production ready с первого коммита

«Потом доделаю» — никогда. Юзер видит пустой экран при ошибке и уходит навсегда.

MUST: error handling, loading state,
empty state для каждого экрана.
MUST NOT: console.log в продакшне,
TODO-комментарии, захардкоженные
секреты в коде.

5

Адаптивность на всех устройствах

60%+ трафика с телефона. Сломанный mobile = потерял больше половины аудитории в день запуска.

MUST: mobile-first. Проверяй на
375px / 768px / 1440px.
Поддержка Chrome, Safari, Firefox.
MUST NOT: фиксированные px-ширины,
overflow без проверки на mobile.

## Слайд 9
**Supabase + Vercel**
Backend + Deploy

Почему Supabase + Vercel для MVP

Полный стек от нуля до прода за $0 — без DevOps и backend-разработчика.

Supabase

PostgreSQL + Auth + Storage + Realtime API — всё в одном. Backend без backend-разработчика.

Плюсы

+

Старт бесплатно — до 500 МБ и 50K запросов/мес

+

Claude пишет SQL — понятный язык, не ORM

+

Auth из коробки — email, Google, GitHub за 10 мин

+

RLS — безопасность данных на уровне БД, не кода

+

MCP-коннектор — Claude работает с базой напрямую

Минусы

−

Vendor lock-in: миграция на другую БД — боль

−

Free лимиты кончаются при росте (~$25/мес потом)

−

Realtime сложнее при 10К+ одновременных соединений

→ Настраивается за час. Claude пишет миграции сам. Не нужен DBA.

Vercel

Хостинг для Next.js: автодеплой, CDN, edge functions, preview на каждый PR.

Плюсы

+

git push → живой URL за 90 секунд

+

Preview URL на каждый PR — показываешь инвесторам до мержа

+

CDN в 100+ странах — быстро для всех

+

Zero-config для Next.js — работает сразу

+

Аналитика и логи в дашборде

Минусы

−

Serverless limits: функция живёт 10 сек на free

−

Дорожает при трафике (Hobby $0 → Pro $20/мес)

−

Vendor lock-in на Next.js — другой фреймворк сложнее

→ Деплой без DevOps. Домен в 2 клика. Инфра сама масштабируется.

При 1000 юзеров — переходи. До 1000 — не думай об этом.

Supabase и Vercel созданы для старта. Когда вырастешь — мигрируешь на AWS/GCP с деньгами и командой. Сейчас — просто строй продукт.

**P1 MVP Scaffolder**
До первой строки кода

Алгоритм старта

Исправить план — 2 минуты. Исправить код — 2 часа. Не пропускай эти 5 шагов.

→2

Включи режим планирования

Claude составляет полный план — без единой строки кода. Видишь всю архитектуру до того, как потратил время.

→ Codex / Antigravity

Читай файлы и корректируй

Читатель думает: «Это про меня. Хочу узнать больше.»

«Я помог [конкретный человек] достичь [конкретный результат]» — работает лучше, чем «Мой продукт делает [фича]»

Сторителлинг — шаг 3

Конфликт = боль клиента

Без конфликта — нет истории. Опиши боль так точно, что читатель думает «это про меня» — и он уже доверяет.

Внешняя боль

Что происходит снаружи. Факт, который можно наблюдать.

## Слайд 19
**Хук + структура + удержание**
Сторителлинг — шаг 6

Хук + структура + удержание

История есть. Вот как упаковать её в пост (FB / LinkedIn / X) и видео (30–45 сек).

5 типов хука — первые 2 строки решают всё

Парадокс

«Построил продукт за 6 месяцев. Никто не купил. Это лучшее, что со мной случилось.»

Цифра + неожиданность

«47 DM. Все без ответа. Инвестор среди них.»

Контр-интуитивно

«Я перестал отвечать клиентам — и продажи выросли.»

Открытый вопрос

«Ты знаешь, сколько денег ты теряешь на каждом непрочитанном сообщении?»

Конкретная сцена

«Вторник. 2 ночи. 47 непрочитанных. Я закрыл ноутбук.»

Структура поста и видео

Хук

1–2 строки

Зацепи. Заставь прочитать второй абзац.

Сцена

3–4 строки

Покажи боль через конкретный момент с деталями.

Инсайт

3–4 строки

Что ты понял? Как это изменило тебя или продукт?

CTA

1 строка

Один вопрос или одно действие. Не три.

Удержание до конца

Открой петлю в хуке («вот что я понял после этого...») — закрой в конце. Читатель дочитывает, чтобы получить ответ.
